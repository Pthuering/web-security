Lösung: Trust Boundary Audit

Vorfall #1 — System Prompt Leakage User → System · S
Nutzer überzeugt Direct-Chat-Bot, den vollständigen System Prompt preiszugeben.
Warum User → System?

Klassischer Direct Prompt Injection. Die einzige Trust Boundary in einem Direct-Chat-System ist zwischen User-Input und System. Der Nutzer injiziert eine Anweisung, die das System Prompt überschreibt. Architektur: Direct Chat — keine externen Datenquellen, keine Tools. Die Angriffsfläche ist minimal, aber diese eine Boundary ist trotzdem verwundbar.

Vorfall #2 — Unsichtbare PDF-Anweisungen Externe Docs → Prompt · C
Manipuliertes PDF mit verstecktem weißem Text wird in die Vector DB aufgenommen.
Warum Externe Docs → Prompt-Kontext?

Das ist RAG Poisoning via Indirect Prompt Injection. Das Dokument wird als Kontext in den Prompt eingefügt, und der versteckte Text wird vom LLM als Instruktion interpretiert. Die Trust Boundary: Externe Dokumente (unkontrolliert) werden als Prompt-Kontext (vertrauenswürdig) behandelt. Die unsichtbare Tinte ist nur das Transportmittel — die Schwachstelle ist die fehlende Trennung zwischen Daten und Instruktionen.

Vorfall #3 — Autonomer Dateiexport LLM → Tool-Aufruf · H
Agent liest eine E-Mail mit versteckter Anweisung und leitet eigenständig vertrauliche Dateien weiter.
Warum LLM → Tool-Aufruf?

Ja, der Auslöser ist eine Indirect Prompt Injection via E-Mail (Docs → Prompt). Aber die primäre Schwachstelle ist die Architektur: Der Agent darf Dateien senden, ohne Bestätigung. Das ist Excessive Agency (OWASP LLM06) — zu viele Berechtigungen, zu wenig Human-in-the-Loop. Die Injection allein wäre harmlos, wenn der Agent nicht eigenständig destruktive Aktionen ausführen könnte.

„Ist das nicht RAG Poisoning (Docs → Prompt)?" Die E-Mail-Injection ist der Trigger, aber nicht die Schwachstelle. Denk so: Wenn der Agent für E-Mail-Versand eine Bestätigung bräuchte, wäre die Injection wirkungslos. Die primäre Trust Boundary ist LLM → Tool, weil dort die Aktion ohne Kontrolle passiert. Die Unterscheidung „was hat den Angriff ausgelöst?" vs. „was hat den Schaden ermöglicht?" ist zentral.

Vorfall #4 — Kompromittierte API-Antwort Tool-Ergebnis → LLM · E
API-Endpunkt liefert Prompt-Injection-Payload in der Antwort zurück.
Warum Tool-Ergebnis → LLM?

Die umgekehrte Richtung von #3: Nicht das LLM ruft ein Tool auf, sondern ein Tool-Ergebnis fließt zurück ins LLM und wird ungeprüft als Kontext behandelt. Das LLM unterscheidet nicht zwischen seiner eigenen Planung und dem injizierten Text aus der API-Antwort. Untrusted Return Data — alles, was von außen zurückkommt, muss als potenziell feindlich behandelt werden.

Verwechslung mit Vorfall #3: Beide involvieren Tools. Die Richtung entscheidet: #3 = LLM ruft Tool (Excessive Agency). #4 = Tool antwortet an LLM (Untrusted Return). #3 ist ein Autorisierungsproblem (darf das LLM das tun?). #4 ist ein Vertrauensproblem (darf das LLM dieser Antwort glauben?).

Vorfall #5 — Fehlklassifikation in der Pipeline Modell A → Modell B · M
Klassifikator-Fehler kaskadiert durch die gesamte Multi-Model-Pipeline.
Warum Modell A → Modell B?

In einer Multi-Model-Pipeline vertraut jedes nachfolgende Modell dem Output des vorherigen. Der Klassifikator labelt die Anfrage falsch → das LLM arbeitet mit dem falschen Kontext → das Safety-Modell prüft gegen den falschen Kontext. Ein Fehler in Modell A kaskadiert durch die gesamte Pipeline. Die primäre Trust Boundary ist dort, wo der fehlerhafte Output als vertrauenswürdiger Input behandelt wird: Modell A → Modell B.

Vorfall #6 — Halluziniertes SQL-UPDATE LLM-Output → Downstream · A
LLM generiert SQL mit unbeabsichtigtem UPDATE-Befehl, der direkt ausgeführt wird.
Warum LLM-Output → Downstream-System?

Der Schlüssel: Es gibt keinen Angreifer. Das LLM halluziniert einen UPDATE-Befehl, und der wird direkt an die Datenbank weitergegeben. Die Schwachstelle ist Insecure Output Handling — der LLM-Output wird als vertrauenswürdiger Code behandelt. Verteidigung: SQL-Output validieren (nur SELECT erlauben), Datenbank-User mit Read-Only-Rechten, oder einen SQL-Parser vorschalten.

„Das ist doch ein Tool-Problem (LLM → Tool)?" Nein — der Unterschied ist subtil aber wichtig. Excessive Agency (LLM → Tool) wäre: Das LLM entscheidet sich, die Datenbank zu modifizieren. Insecure Output wäre: Das LLM generiert Output, der ohne Intent schädlich ist (Halluzination), und ein Downstream-System führt ihn blind aus. Hier ist es Insecure Output, weil die Halluzination der Auslöser ist, nicht eine bewusste Agentur.

🤔 Reflexion

Trigger vs. Schwachstelle: Vorfall #3 ist die beste Übung für diese Unterscheidung. Die E-Mail-Injection ist der Trigger, Excessive Agency ist die Schwachstelle. In Security denkt man immer in zwei Fragen: Wie kommt der Angreifer rein? (Attack Vector) und Warum richtet er Schaden an? (Vulnerability). Die Verteidigung adressiert die Schwachstelle, nicht den Trigger.
Die Richtung zählt: Wenn du #3 und #4 verwechselt hast — das ist ein häufiger Fehler. Zeichne dir die Tool-Use-Pipeline als Pfeil-Diagramm: LLM → Tool (Aktion) vs. Tool → LLM (Daten zurück). Jede Richtung hat ein eigenes Risikoprofil und andere Verteidigungsstrategien.
Halluzination als Security-Problem: Vorfall #6 zeigt: Man braucht keinen Angreifer für einen Sicherheitsvorfall. LLMs halluzinieren — und wenn ihr Output direkt ausgeführt wird, ist eine Halluzination ein Exploit ohne Angreifer. Das ist der Grund, warum Insecure Output Handling im OWASP Top 10 steht.

🔄 Transfer-Fragen

Für dein Ticketing-System: Wenn du einen AI-Assistenten baust, der Tickets erstellen kann — welches Architekturmuster ist das? Welche Trust Boundaries hast du, und welche sind die kritischsten? Zeichne das Datenfluss-Diagramm.
Querverweis Web-Track: Die Trust-Boundary-Analyse hier ist das Äquivalent von Netzwerk-Segmentierung im Web-Track (Modul 4). In beiden Fällen geht es um die Frage: Welche Komponente darf mit welcher kommunizieren, und in welche Richtung? Same principle, different domain.
