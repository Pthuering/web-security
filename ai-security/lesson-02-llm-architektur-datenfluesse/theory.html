LLM-Architektur & Datenflüsse

🔄 Spiral-Review: Modul 1

Drei Fragen aus der letzten Lektion. Beantworte sie im Kopf, bevor du aufdeckst.

Nenne die zwei grundlegenden Pipelines eines AI-Systems und ihren Hauptunterschied für Security.

Klick zum Aufdecken

Training Pipeline (Daten & Modell schützen — langsam, persistent) und Inference Pipeline (Input & Output kontrollieren — schnell, transient). Training verändert das Modell, Inference nutzt es.

Was ist der Unterschied zwischen Data Poisoning und RAG Poisoning?

Klick zum Aufdecken

Data Poisoning manipuliert Trainingsdaten → das Modell selbst wird korrumpiert (Training-Phase). RAG Poisoning manipuliert abrufbare Dokumente → das Modell bleibt intakt, aber sein Kontext wird zur Laufzeit manipuliert (Inference-Phase).

Was macht der Tokenizer, und warum ist er security-relevant?

Klick zum Aufdecken

Der Tokenizer zerlegt Text in Sub-Wort-Einheiten (Tokens). Er ist security-relevant, weil er deterministisch und vorhersagbar ist — Angreifer können Unicode-Tricks nutzen, um Keyword-Filter zu umgehen, indem sie die Tokenisierung manipulieren.

1. Von der Pipeline zur Architektur

In Modul 1 hast du die Pipeline kennengelernt: der lineare Datenfluss von Input zu Output. Aber echte LLM-Anwendungen sind selten linear — sie haben Schleifen, externe Datenquellen und Tool-Aufrufe.

Diese Lektion stellt dir fünf Architekturmuster vor — von einfach bis komplex. Jedes Muster erweitert die Angriffsfläche um neue Komponenten und Trust Boundaries.

Zentrales Konzept: Trust Boundary

Eine Trust Boundary ist die Grenze zwischen Komponenten mit unterschiedlichem Vertrauensniveau. Jedes Mal, wenn Daten eine Trust Boundary überschreiten, entsteht ein potenzieller Angriffspunkt. Faustregel: Je mehr Trust Boundaries eine Architektur hat, desto größer die Angriffsfläche.

1
Direct Chat
Das einfachste Muster — und die kleinste Angriffsfläche
Pattern 1: Direct Chat — eine Trust Boundary (User → LLM)

System Prompt (vom Entwickler) + User Prompt → LLM → Response. Eine einzelne Trust Boundary: der User-Input. Risiken: Direct Prompt Injection und System Prompt Leakage.

⚡ 1 Trust Boundary · Risiko: Prompt Injection, Prompt Leakage
2
RAG (Retrieval-Augmented Generation)
Externe Dokumente als Kontext — eine zweite Datenquelle, eine zweite Trust Boundary
Pattern 2: RAG — zwei Trust Boundaries (User-Input + externe Dokumente)

Der User-Query wird zu einem Vektor, findet ähnliche Dokumente in der Vector DB, und diese werden als Kontext in den Prompt eingefügt. Neue Trust Boundary: Die Dokumente in der Vector DB. Wer kontrolliert, was dort reinkommt?

Dokumente aus der Vector DB werden dem LLM als Kontext präsentiert — das Modell unterscheidet nicht zwischen System Prompt und injiziertem Dokument-Content. Ein manipuliertes Dokument kann eine Indirect Prompt Injection enthalten.

⚡ 2 Trust Boundaries · Risiko: Indirect Prompt Injection, RAG Poisoning
3
Tool Use / Function Calling
Das LLM kann handeln — Code ausführen, APIs aufrufen, Daten verändern
Pattern 3: Tool Use — das LLM entscheidet, welches Tool aufgerufen wird

Der qualitative Sprung: Das LLM liest nicht nur, es handelt. Es entscheidet, welches Tool mit welchen Parametern aufgerufen wird. Das Ergebnis fließt zurück ins LLM.

// Beispiel: Tool-Definition für ein LLM
const tools = [{
  name: "send_email",
  description: "Sendet eine E-Mail an die angegebene Adresse",
  parameters: {
    to: { type: "string" },
    subject: { type: "string" },
    body: { type: "string" }
  }
}];

// ⚠ Was wenn der User das LLM überzeugt,
// eine E-Mail an einen Angreifer zu senden?
// ⚠ Was wenn das Tool-Ergebnis manipulierten Text enthält?
Kritische Erkenntnis

Tool Use verwandelt Prompt Injection von einem Informationsproblem (LLM sagt etwas Falsches) in ein Aktionsproblem (LLM tut etwas Schädliches). Ein Chatbot, der lügt, ist ärgerlich. Ein Agent, der deine Datenbank löscht, ist katastrophal.

⚡ Kritisch · Risiko: Excessive Agency, Tool Misuse, Privilege Escalation
4
Agent Loop (ReAct-Pattern)
Das LLM plant, handelt, beobachtet, plant erneut — in einer Schleife
Pattern 4: Agent Loop — Think → Act → Observe → Repeat

Das ReAct-Pattern (Reason + Act): Das LLM denkt nach, führt ein Tool aus, beobachtet das Ergebnis, und entscheidet dann, ob es weitergeht oder fertig ist. Der kritische Unterschied zu Tool Use: Die Schleife. Jede Iteration kann auf dem Ergebnis der vorherigen aufbauen — ohne menschliche Kontrolle dazwischen.

Eskalationspotenzial

Agent Loops können eskalieren: In Iteration 1 liest der Agent eine Datei. In Iteration 2 extrahiert er daraus eine URL. In Iteration 3 ruft er die URL auf. Kein einzelner Schritt ist gefährlich — aber die Kette kann es sein. Das ist Excessive Agency (OWASP LLM06): zu viel Autonomie, zu viele Berechtigungen, zu wenig Kontrolle.

⚡ Höchstes Risiko · Eskalation, Autonomie ohne Kontrolle, Compound Actions
5
Multi-Model Pipeline
Mehrere Modelle in Reihe — Output von Modell A ist Input für Modell B
Pattern 5: Multi-Model — jede Modellgrenze ist eine Trust Boundary

Beispiel: Ein Klassifikator entscheidet, ob eine Anfrage erlaubt ist → ein LLM generiert die Antwort → ein Safety-Filter prüft die Ausgabe. Jede Modellgrenze ist eine Trust Boundary, weil der Output von Modell A der Input von Modell B ist — genau das Muster von Insecure Output Handling.

⚡ Kaskadierend · Risiko: Output eines Modells injiziert das nächste

6. Trust Boundaries — das vereinheitlichende Prinzip

Alle fünf Muster folgen demselben Grundprinzip: Wo Daten eine Vertrauensgrenze überschreiten, entsteht ein Risiko. Die Muster unterscheiden sich nur in der Anzahl und Art der Grenzen:

Trust Boundaries pro Architekturmuster — mehr Grenzen = größere Angriffsfläche
Zeichne diese Übersicht in dein Notebook. „Architekturkomplexität = Anzahl Trust Boundaries × Autonomiegrad. Agent Loops maximieren beides."

📋 Worked Example: Microsoft Copilot & Tool Misuse (2024)

Security-Researcher Bargury demonstrierte 2024, wie Microsoft 365 Copilot — ein Tool-Use-Agent mit Zugang zu E-Mail, Kalender und Dateien — durch Indirect Prompt Injection manipuliert werden kann.

Angriffsvektor
  • Versteckte Anweisungen in einer E-Mail (Indirect Prompt Injection via RAG)
  • Copilot liest die E-Mail als Kontext → führt eingebettete Anweisung aus
  • Tool Use erlaubt: E-Mails senden, Dateien lesen, Kalender bearbeiten
  • Architektur-Pattern: RAG + Tool Use + Agent Loop — maximale Angriffsfläche
Verteidigung
  • Least Privilege: Tools nur die minimal nötigen Berechtigungen geben
  • Human-in-the-Loop: Bestätigung vor destruktiven Aktionen
  • Input-Sanitization: Dokument-Content als Daten behandeln, nicht als Instruktion
  • Monitoring: Tool-Aufrufe loggen und auf Anomalien prüfen
Pattern-Erkennung

Drei Patterns kombiniert: RAG (E-Mails als Kontext) + Tool Use (E-Mail senden) + Agent Loop (mehrere Schritte autonom). Genau so entstehen die gefährlichsten Szenarien — nicht durch eine einzelne Schwachstelle, sondern durch die Kombination von Architekturmustern.

📝 Zusammenfassung

Du kennst jetzt die fünf Architekturmuster moderner LLM-Anwendungen und kannst sie anhand ihrer Trust Boundaries unterscheiden. Das Schlüsselkonzept: Jede Trust Boundary ist ein potenzieller Angriffspunkt. In den folgenden Modulen wirst du für jede Boundary lernen, wie Angriffe konkret funktionieren und wie man sie verteidigt.
