AI/ML-Grundlagen für Security

🔄 Vorwissen-Check

Erste Lektion im KI-Track — kein Recall nötig. Stattdessen: Überprüfe dein Vorwissen. Klick zum Aufdecken.

Was ist der Unterschied zwischen "Training" und "Inference" bei einem ML-Modell?

Klick zum Aufdecken

Training = Das Modell lernt aus Daten (Gewichte werden angepasst). Inference = Das fertige Modell wird benutzt (Gewichte sind fix, neue Eingaben werden verarbeitet). Wenn du das schon wusstest — perfekt. Wenn nicht, lernst du es gleich.

Was bedeutet "Angriffsfläche" (Attack Surface)?

Klick zum Aufdecken

Die Summe aller Punkte, an denen ein Angreifer mit einem System interagieren kann. Je größer die Angriffsfläche, desto mehr potenzielle Schwachstellen. Genau das werden wir hier auf AI/ML-Systeme anwenden.

1. Die Landkarte: ML → DL → NLP → LLMs

Bevor wir Angriffsflächen verstehen können, brauchen wir eine mentale Landkarte. Diese Begriffe sind keine Synonyme — sie sind verschachtelte Kategorien:

ML-Taxonomie: LLMs sind eine Teilmenge von NLP ⊂ Deep Learning ⊂ Machine Learning
Security-Relevanz

Jede Schicht hat eigene Angriffsflächen. Ein Angriff auf ML-Ebene (z.B. Data Poisoning) betrifft auch alle Unterschichten. Ein Angriff auf LLM-Ebene (z.B. Prompt Injection) ist spezifisch für Sprachmodelle. Im OWASP LLM Top 10 fokussieren wir uns auf die LLM-Schicht — aber das Verständnis der Gesamtstruktur hilft, die Grenzen dieser Perspektive zu erkennen.

Schreibe in dein Notebook: „ML ⊃ DL ⊃ NLP ⊃ LLMs — Angriffe auf höherer Ebene betreffen auch tiefere Schichten."

2. Transformer-Architektur (Security-Perspektive)

LLMs basieren auf der Transformer-Architektur (2017). Du musst kein Mathematiker sein, um Security zu machen — aber du musst verstehen, wo Daten fließen und wo Angreifer eingreifen können.

Was ein Transformer tut (vereinfacht)

Ein Transformer nimmt eine Sequenz von Tokens (Wort-Fragmenten) als Eingabe und berechnet, welches Token als nächstes am wahrscheinlichsten kommt. Das Geheimnis: der Attention-Mechanismus — jedes Token „schaut" auf alle anderen Tokens und gewichtet ihre Relevanz.

🔤
Tokenization
Text wird in Sub-Wort-Einheiten zerlegt. „Sicherheitslücke" → [„Sicher", „heits", „lücke"]. Der Tokenizer ist die erste Verarbeitungsstufe.
📐
Embeddings
Jedes Token wird zu einem Zahlenvektor (z.B. 4096 Dimensionen). Ähnliche Wörter → ähnliche Vektoren. Basis für semantische Suche und RAG.
🔍
Attention
„Welche anderen Tokens sind für dieses Token relevant?" Erlaubt dem Modell, Kontext über lange Distanzen zu erfassen.
Angriffsfläche: Tokenizer

Der Tokenizer ist deterministisch und vorhersagbar. Angreifer können Texte so konstruieren, dass die Tokenisierung unerwartete Ergebnisse liefert — z.B. um Filter zu umgehen. Wenn ein Sicherheitsfilter nach dem Wort „Bombe" sucht, aber der Tokenizer „B" + „ombe" erzeugt, greift der Filter möglicherweise nicht.

3. Training vs. Inference Pipeline

Die zentrale Erkenntnis: Ein AI-System hat zwei fundamental verschiedene Betriebsmodi, und jeder hat andere Angriffsflächen.

Training Pipeline (oben) vs. Inference Pipeline (unten) — mit typischen Angriffspunkten
Kernunterschied für Security

Training: Angreifer manipulieren die Daten oder stehlen das Modell. Der Angriff wirkt langfristig und ist schwer zu erkennen.
Inference: Angreifer manipulieren die Eingabe (Prompt) oder nutzen die Ausgabe als Waffe. Der Angriff ist kurzfristig, aber sofort wirksam.

Schreibe in dein Notebook: „Training = Daten & Modell schützen (langsam, persistent). Inference = Input & Output kontrollieren (schnell, transient)."

4. Embeddings, Vector DBs & RAG

Moderne LLM-Anwendungen bestehen nicht nur aus dem Modell. Die häufigste Erweiterung ist RAG (Retrieval-Augmented Generation) — und sie bringt eigene Angriffsflächen mit.

Wie RAG funktioniert

Statt das Modell mit allem Wissen zu trainieren, werden relevante Dokumente zur Laufzeit gefunden und dem Prompt beigefügt:

// Vereinfachter RAG-Ablauf:

// 1. User stellt Frage
const frage = "Was ist unsere Rückgabepolitik?";

// 2. Frage → Embedding (Zahlenvektor)
const frageVektor = embed(frage); // [0.12, -0.34, 0.87, ...]

// 3. Ähnlichste Dokumente aus Vector DB finden
const relevanteDocs = vectorDB.search(frageVektor, topK=3);

// 4. Dokumente + Frage → LLM
const prompt = `Kontext: ${relevanteDocs}\n\nFrage: ${frage}`;
const antwort = llm.generate(prompt);

// ⚠ Was wenn relevanteDocs manipuliert wurde?
// ⚠ Was wenn der Kontext Prompt-Injection-Payload enthält?
Angriffsfläche: RAG Pipeline

RAG-Systeme sind besonders anfällig, weil sie zwei Angriffsflächen kombinieren: Die Vector-DB (wer kontrolliert die Dokumente?) und den Prompt (injizierter Kontext wird wie eine Instruktion behandelt). Das ist Indirect Prompt Injection — ein zentrales Thema in Modul 5.

5. Attack Surface Map: Das Gesamtbild

Jetzt setzen wir alles zusammen. Jede Stufe der AI/ML-Pipeline hat spezifische Angriffsflächen. Diese Tabelle ist dein Referenz-Schema für den gesamten KI-Security-Track:

📋 Worked Example: Samsung & ChatGPT (2023)

Im April 2023 nutzten Samsung-Ingenieure ChatGPT, um internen Quellcode zu debuggen und Meeting-Protokolle zusammenzufassen. Problem: Die eingegebenen Daten wurden Teil der OpenAI-Trainingsdaten.

Angriffsvektor
  • Kein aktiver Angreifer nötig — die Mitarbeiter selbst gaben vertrauliche Daten in ein externes System ein
  • Pipeline-Stufe: Trainingsdaten — die Prompts wurden zu Trainingsdaten für zukünftige Modelle
  • Angriffsfläche: Die Grenze zwischen „Inference nutzen" und „Training füttern" war unklar
Verteidigung
  • Samsung verbot ChatGPT für alle Mitarbeiter
  • Langfristig: Data Loss Prevention (DLP) Policies, Unternehmens-eigene LLM-Instanzen, Opt-out aus Training
  • OWASP LLM02 (Sensitive Information Disclosure) adressiert genau dieses Szenario
Pattern-Mismatch

Hier gab es keinen „Hacker". Die Schwachstelle war ein Verständnisproblem: Mitarbeiter verstanden nicht, dass „Inference nutzen" gleichzeitig „Training füttern" bedeuten kann. Genau deswegen brauchst du die Pipeline-Diagramme oben.
