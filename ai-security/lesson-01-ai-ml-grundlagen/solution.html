KI-Security Track

Lösung: Pipeline-Angriffskartierung

Vorfall #1 — Gefälschte Reviews Trainingsdaten → T
Tausende gefälschte Reviews werden in den Crawling-Datensatz eingeschleust, der als Trainingsgrundlage dient.
Warum Trainingsdaten?

Der Angriff verändert die Datenbasis, aus der das Modell lernt. Das ist klassisches Data Poisoning (OWASP LLM04). Die Wirkung entfaltet sich erst beim nächsten Training — langsam, persistent und schwer zu erkennen.

Vorfall #2 — Gestohlene Gewichtsdateien Modellgewichte → E
Ein Ex-Mitarbeiter kopiert die vollständigen Modellgewichte und veröffentlicht ein Klon-Modell.
Warum Modellgewichte?

Die Gewichte sind das Modell. Wer die Gewichte hat, kann das Modell replizieren. Unter Supply Chain (OWASP LLM03), weil gestohlene Modelle oft unter falscher Provenance weiterverbreitet werden.

Vorfall #3 — „Ignoriere alle Anweisungen" User-Prompt → N
Ein Nutzer gibt eine manipulierte Eingabe, die das System Prompt überschreibt.
Warum User-Prompt?

Das ist der Lehrbuch-Fall von Direct Prompt Injection (OWASP LLM01). Der Angriffspunkt ist die User-Eingabe in der Inference-Pipeline. Das Modell behandelt die Anweisung des Nutzers als gleichwertig mit (oder sogar vorrangig gegenüber) dem System Prompt.

Vorfall #4 — Unicode-Manipulation Tokenizer → S
Unsichtbare Unicode-Zeichen werden eingesetzt, um die Textzerlegung zu manipulieren und einen Keyword-Filter zu umgehen.
Warum Tokenizer?

Der Angriff zielt nicht auf das Modell selbst, sondern auf den Verarbeitungsschritt davor. Der Tokenizer zerlegt den Text in Fragmente — und wenn diese Fragmente nicht den gesperrten Mustern entsprechen, greift der Filter nicht. Der Mensch liest „Waffe", der Tokenizer sieht [„Wa", „​", „ffe"] (mit unsichtbarem Zero-Width-Joiner).

Verwechslungsgefahr mit Prompt Injection: Prompt Injection manipuliert den Inhalt der Eingabe (was gesagt wird). Adversarial Tokens manipulieren die Darstellung (wie es zerlegt wird). Der Unterschied: Bei Prompt Injection versteht das Modell die Anweisung — bei Tokenizer-Angriffen „sieht" das Modell bzw. der Filter etwas anderes als der Mensch.

Vorfall #5 — SQL aus LLM-Output Modell-Output → O
Das LLM generiert SQL-Queries, die ohne Sanitization an die Datenbank weitergegeben werden.
Warum Modell-Output?

Das Problem ist nicht, was ins Modell reingeht, sondern was mit der Ausgabe passiert. Der Output des LLMs wird als vertrauenswürdiger Code behandelt und direkt ausgeführt — Insecure Output Handling (OWASP LLM05). Die Verteidigung ist Output-Validation, nicht Input-Filtering.

„Aber der Angreifer tippt doch den Prompt ein — ist das nicht Prompt Injection?" Ja, der Trigger kommt über den Prompt. Aber die Schwachstelle liegt im Output-Handling: Selbst ohne bösen Nutzer könnte ein Halluzinations-bedingtes DROP TABLE Schaden anrichten. Prompt Injection ist der Angriffsvektor, Insecure Output ist die Schwachstelle.

Vorfall #6 — Manipuliertes HR-Dokument Vector DB / RAG → R
Ein manipuliertes Dokument mit versteckten Anweisungen wird in die Wissensdatenbank hochgeladen.
Warum Vector DB / RAG?

Das Dokument wird nicht zum Training verwendet — es wird zur Laufzeit als Kontext in den Prompt eingefügt. Das ist RAG Poisoning (OWASP LLM08), eine Form von Indirect Prompt Injection. Der Angriff nutzt das RAG-System als Transportmechanismus für bösartige Anweisungen.

Verwechslungsgefahr mit Data Poisoning: Data Poisoning verändert die Trainingsdaten → das Modell selbst wird korrumpiert. RAG Poisoning verändert die abrufbaren Dokumente → das Modell bleibt intakt, aber sein Kontext wird manipuliert. Zeitpunkt: Training vs. Inference.

🤔 Reflexion

Was war der schwierigste Vorfall? — Für die meisten ist #4 (Tokenizer) am schwierigsten, weil er wie Prompt Injection aussieht. Der entscheidende Unterschied: Der Angriff manipuliert nicht was gesagt wird, sondern wie es verarbeitet wird.
Wo hast du zuerst falsch gedacht? — Wenn du #5 als „Prompt Injection" eingestuft hast: Das ist nachvollziehbar, weil der Trigger tatsächlich über den Prompt kommt. Die Unterscheidung „Angriffsvektor vs. Schwachstelle" ist subtil aber wichtig. Prompt Injection ist der wie, Insecure Output ist das wo.
Welches Muster zieht sich durch? — Die Pipeline hat eine klare Richtung: Daten → Verarbeitung → Output. Jede Stufe vertraut der vorherigen. Übermäßiges Vertrauen zwischen Pipeline-Stufen ist das wiederkehrende Security-Problem.

🔄 Transfer-Fragen

Für dein Ticketing-System: Wenn du einen AI-gestützten Chatbot für Ticket-Erstellung einbaust — an welchen Pipeline-Stufen müsstest du Security-Controls einbauen? Denk an alle sechs Stufen.
Querverweis Web-Track: SQL Injection (Web-Modul 7) und Insecure Output Handling (Vorfall #5) folgen demselben Muster: Output eines Systems wird als vertrauenswürdiger Input eines anderen Systems behandelt. Wo noch begegnet dir dieses Pattern?
Training vs. Inference als Denkmodell: Der Samsung-Vorfall aus der Theorie zeigt: Die Grenze zwischen „Inference nutzen" und „Training füttern" kann verschwimmen. Gilt das auch für deine eigene Nutzung von Claude oder ChatGPT? Was gibst du ein, das du nicht als Trainingsdaten freigeben würdest?
