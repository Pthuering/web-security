Lösung: Reflected XSS

Wie das Passwort gefunden wird — und warum der Fix funktioniert.

⚠ Spoiler

Diese Seite enthält die vollständige Lösung. Nur lesen, wenn du das Rätsel gelöst hast oder nicht weiterkommst.

Schritt-für-Schritt-Lösungsweg

1. Die Schwachstelle identifizieren

Die Suchfunktion von NeonShop nimmt den User-Input und fügt ihn via innerHTML direkt ins DOM ein:

Verwundbarer Code
output.innerHTML = `Suchergebnisse für: <strong>${query}</strong>`;
// ⚠️ query enthält unescapten User-Input

Das bedeutet: Alles, was du in das Suchfeld eingibst, wird als HTML interpretiert, nicht als reiner Text.

2. HTML Injection testen

Ein erster Test — gib in die Suche ein:

<b>Test</b>

Wenn "Test" fettgedruckt erscheint, ist HTML Injection bestätigt. Der Browser interpretiert deinen Input als HTML-Markup.

3. JavaScript-Ausführung (XSS)

Der nächste Schritt: JavaScript ausführen. Ein direkter <script>-Tag funktioniert nicht über innerHTML (Browser blocken das). Aber Event-Handler funktionieren:

<img src=x onerror="alert(1)">

Dieser Payload erzeugt ein <img>-Element, das versucht, das Bild x zu laden. Das schlägt fehl, und der onerror-Handler feuert — mit beliebigem JavaScript.

4. Das versteckte Passwort extrahieren

Im JavaScript-Code der Seite ist eine globale Variable SECRET_KEY definiert. Da dein XSS-Payload im Kontext der Seite läuft, kann er auf alle globalen Variablen zugreifen:

<img src=x onerror="alert(SECRET_KEY)">

Das zeigt einen Alert mit dem Passwort: xss-m4ster-2024

Alternative Payloads

Es gibt viele Wege zum Ziel. Einige Varianten:

// Alert
<img src=x onerror="alert(SECRET_KEY)">

// Direkt in die Seite schreiben
<img src=x onerror="document.body.innerText=SECRET_KEY">

// In die Konsole loggen
<img src=x onerror="console.log(SECRET_KEY)">

// SVG-basiert (anderer Element-Typ)
<svg onload="alert(SECRET_KEY)">

Warum funktioniert das?

Die Schwachstelle hat zwei Zutaten:

1. Unkontrollierter Input → HTML-Kontext: innerHTML behandelt den String nicht als Daten, sondern als HTML-Code. Der Browser kann nicht unterscheiden, ob das <img>-Tag vom Entwickler stammt oder vom Angreifer eingeschleust wurde.

2. JavaScript hat vollen Kontext-Zugriff: Einmal ausgeführt, hat der eingeschleuste Code denselben Zugriff wie jedes andere Script auf der Seite — Variablen, Cookies, DOM, alles. In einem realen Szenario könnte ein Angreifer Session-Cookies stehlen, Formulare manipulieren oder den Nutzer auf eine Phishing-Seite umleiten.

Der Fix

Vergleiche den verwundbaren Code mit der sicheren Version:

❌ VERWUNDBAR
output.innerHTML =
  `Ergebnisse für: ${query}`;
✅ SICHER
output.textContent =
  `Ergebnisse für: ${query}`;

textContent behandelt den String immer als reinen Text. Selbst wenn der User <script> eingibt, wird es als die Buchstaben <, s, c, r, ... angezeigt — nicht als HTML interpretiert.

Wenn du HTML-Struktur brauchst (z.B. den Suchbegriff fett machen), verwende eine Escape-Funktion:

Sicherer Code mit HTML-Struktur
function escapeHtml(str) {
  const div = document.createElement('div');
  div.textContent = str;
  return div.innerHTML;
}

// Jetzt sicher:
output.innerHTML = `Ergebnisse für: <strong>${escapeHtml(query)}</strong>`;
Merke: Die Escape-Regel

Daten und Code müssen immer getrennt bleiben. Wenn User-Input in einen Kontext eingefügt wird, der Code ausführen kann (HTML, SQL, Shell...), muss er vorher so transformiert werden, dass er nur als Daten interpretiert werden kann. Das ist das grundlegendste Sicherheitsprinzip im Web — und kommt in fast jeder Lektion in abgewandelter Form wieder.

Reflexion

Häufige Hürden bei dieser Übung:

  • "Ich hab <script> probiert und nichts passierte" — Moderne Browser blockieren Script-Tags via innerHTML. Das ist kein Schutz gegen XSS, weil Event-Handler (onerror, onload) trotzdem funktionieren.
  • "Ich wusste nicht, dass ich auf SECRET_KEY zugreifen kann" — Jedes JavaScript auf einer Seite teilt denselben globalen Scope. Eingeschleuster Code hat vollen Zugriff auf alle Variablen, Cookies und das DOM.
  • "Ich hab nicht an Event-Handler gedacht" — XSS-Payloads nutzen selten <script>-Tags. In der Praxis sind Event-Handler, SVG, und andere HTML-Features die gängigen Vektoren.

Transfer

Überall, wo User-Input angezeigt wird:

  • Suchfunktionen: "Sie suchten nach: ..." — der Klassiker.
  • Profilseiten: Benutzername, Bio, Statusnachrichten.
  • Kommentare / Chat: Jede User-generierte Textanzeige.
  • Fehlerseiten: "Die Seite /BENUTZEREINGABE wurde nicht gefunden."
  • URL-Parameter in Dashboards: Filter, Sortierung, Tabs, die aus der URL gelesen und angezeigt werden.

Faustregel: Jedes Mal, wenn du innerHTML, document.write(), oder Template Literals mit User-Input in einem HTML-Kontext verwendest — aufpassen.

Ausblick

In der nächsten Lektion schauen wir uns DOM-based XSS an — eine Variante, bei der der Server gar nicht beteiligt ist. Das Prinzip ist dasselbe (Daten → Code), aber der Angriffsvektor und die Verteidigung sehen anders aus.

Verbindung zu AI-Security: Das Muster "unvertrauenswürdiger Input wird als Instruktion interpretiert" ist identisch mit Prompt Injection bei LLMs. Wenn du XSS verstehst, hast du die Intuition für Prompt Injection bereits.
