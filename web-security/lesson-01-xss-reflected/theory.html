Reflected XSS — Theorie

Wie unsanitisierte Eingaben den Browser zum Angreifer machen.

📎 Spiral-Review

Dies ist die erste Lektion — kein Review nötig. Ab Lesson 02 starten hier 2–3 Recall-Fragen zu vorherigen Konzepten.

Was ist XSS?

Cross-Site Scripting (XSS) ist eine der häufigsten und ältesten Schwachstellen im Web. Das Grundprinzip:

Kernproblem

Ein Angreifer schleust eigenen JavaScript-Code in eine Webseite ein, die dann im Browser eines anderen Nutzers ausgeführt wird — mit allen Rechten, die dieser Nutzer auf der Seite hat.

Warum ist das gefährlich? Weil JavaScript im Browser alles kann, was der Nutzer kann: Cookies auslesen (Session-Hijacking), Formulare manipulieren, Tastatureingaben mitschneiden, Umleitungen erzwingen, sogar Krypto-Miner starten.

Die drei XSS-Varianten

XSS gibt es in drei Geschmacksrichtungen — diese Lektion behandelt die erste:

DIESE LEKTION
Reflected XSS

Der Schadcode steckt in der URL und wird vom Server direkt in die Antwort reflektiert. Nicht gespeichert — einmalig.

KOMMT SPÄTER
Stored XSS

Der Schadcode wird auf dem Server gespeichert (z.B. in einem Kommentar) und trifft jeden Besucher.

KOMMT SPÄTER
DOM-based XSS

Der Server ist nicht beteiligt — das clientseitige JavaScript verarbeitet Input unsicher und manipuliert das DOM direkt.

Wie funktioniert Reflected XSS?

Der Angriff folgt einem klaren Muster. Stell dir eine Suchseite vor, die den Suchbegriff in der URL annimmt und ihn auf der Ergebnisseite anzeigt:

// Die URL
https://shop.example.com/search?q=Laptop

// Was der Server zurückgibt (HTML)
<h2>Suchergebnisse für: Laptop</h2>

Soweit harmlos. Aber was passiert, wenn jemand statt "Laptop" etwas anderes eingibt?

// Manipulierte URL
https://shop.example.com/search?q=<script>alert('XSS')</script>

// Was der Server zurückgibt
<h2>Suchergebnisse für: <script>alert('XSS')</script></h2>

Der Browser sieht ein <script>-Tag und führt es aus. Er kann nicht unterscheiden, ob das Script vom Entwickler stammt oder vom Angreifer eingeschleust wurde.

Der Angriffsflow — Schritt für Schritt

Das Entscheidende: Der Server speichert nichts. Der Schadcode existiert nur in der URL und wird einmalig reflektiert. Deshalb "Reflected" — der Input wird wie ein Spiegel zurückgeworfen.

Worked Example: Verwundbarer Suchcode

Hier ein typisches verwundbares Pattern — eine Suchseite, die den Suchbegriff aus der URL liest und anzeigt:

HTML + JS — verwundbar
<div id="results"></div>

<script>
  // URL-Parameter auslesen
  const params = new URLSearchParams(window.location.search);
  const query = params.get('q');

  // ⚠️ GEFÄHRLICH: User-Input direkt als HTML einfügen
  document.getElementById('results').innerHTML =
    `Ergebnisse für: ${query}`;
</script>
⚠ Das Problem

innerHTML interpretiert den String als HTML. Wenn query ein <script>-Tag enthält, wird es als echtes Script-Element ins DOM eingefügt. Der Browser unterscheidet nicht zwischen "gewolltem" und "eingeschleustem" HTML.

Warum innerHTML + User-Input = Gefahr

Das Kernproblem ist ein Kontextwechsel: Der Wert in query ist ein Daten-String. Aber innerHTML interpretiert ihn als HTML-Code. Diese Grenze zwischen Daten und Code ist genau die Stelle, an der XSS passiert.

Übrigens: Moderne Browser blockieren direkte <script>-Tags, die über innerHTML eingefügt werden. Aber es gibt viele Umwege — z.B. über Event-Handler:

Payload ohne <script>-Tag
<img src=x onerror="alert('XSS')">

// Der Browser versucht das Bild "x" zu laden,
// scheitert, und führt den onerror-Handler aus.

Verteidigung: Output Encoding

Die Lösung ist konzeptionell einfach: Sorge dafür, dass User-Input niemals als Code interpretiert wird.

Fix 1: textContent statt innerHTML

JS — sicher
// ✅ textContent behandelt ALLES als reinen Text
document.getElementById('results').textContent =
  `Ergebnisse für: ${query}`;

// Selbst <script>alert(1)</script> wird als
// harmloser Text angezeigt, nicht ausgeführt.

Fix 2: HTML-Entities escapen

Wenn du HTML-Struktur brauchst (z.B. den Suchbegriff fett darstellen willst), musst du den User-Input encoden, bevor er ins HTML kommt:

JS — Escape-Funktion
function escapeHtml(str) {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#039;');
}

// Jetzt sicher mit innerHTML verwendbar:
results.innerHTML = `Ergebnisse für: <b>${escapeHtml(query)}</b>`;
Merke

Output Encoding bedeutet: Sonderzeichen (<, >, &, ", ') werden in ihre harmlosen HTML-Entity-Äquivalente umgewandelt, bevor sie in den HTML-Kontext kommen. So bleiben sie Daten und werden nie zu Code.

Zusammenfassung — Kernbegriffe

XSS

Einschleusen von Script-Code in fremde Webseiten

Reflected

Payload in der URL, einmalig vom Server reflektiert

innerHTML

Interpretiert Strings als HTML — gefährlich mit User-Input

Output Encoding

Sonderzeichen escapen, damit Daten nie zu Code werden

Payload

Der ausführbare Teil eines Angriffs

textContent

Sichere Alternative — behandelt alles als reinen Text
